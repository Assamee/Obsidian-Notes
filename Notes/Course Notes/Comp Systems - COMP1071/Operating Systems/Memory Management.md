## Intro to Memory Management
### The Role of the OS in Memory Management
- **Tracking:** The OS keeps track of which parts of memory are currently in use and which are free
- **Allocation & Deallocation:** It allocates free memory to processes when they need it and deallocates it when they are finished
- **Data Movement:** It decides which processes (or parts of processes) and data need to be moved into and out of memory, handling the transfer between RAM and the physical disk
---
### Hardware Protection:
#### Necessity of Protection:
- **Memory Isolation:** Memory must be protected so that one process cannot interfere with the memory space of another
- **Execution Safety:** Each process is allocated a separate memory space, which prevents processes from accidentally or maliciously modifying the data or code of other active tasks
#### Base & Limits Registers:
- **Base Register:** Stores the physical memory address of the start of a process's allotted memory space
- **Limit Register:** Stores the size (or range) of the logical allotted address space for a process
_**Note:** A process can access addresses starting from $base$ up to $base + limit$_
##### Hardware Address Checking
- For every memory access generated in user mode, the CPU must check that the address falls within the legal range
---
### Addressing & The MMU
#### Logical & Physical Addressing
- **Logical Address (Virtual Address):** An address generated by the CPU during execution, it is also referred to as a **virtual address**
- **Physical Address:** The actual address seen by the memory unit (RAM) when data is loaded or stored
- **Explanation:** A **logical address** is like a relative coordinate (e.g., "the 3rd house on my street"), while a **physical address** is the absolute coordinate used by the post office (e.g., "1024 Memory Lane")
- **User Isolation:** Application programs only ever deal with logical addresses, they never know the physical addresses in RAM
#### The Memory Management Unit (MMU)
- **Memory Management Unit (MMU):** A hardware device that handles the run-time mapping (translation) of virtual/logical addresses to physical addresses
- **Relocation Register:** In a simple mapping scheme, the **Base register** is renamed the **Relocation Register**. Its value is added to every address generated by a user process at the time it is sent to memory
#### Simple Relocation Calculation Example
1. The CPU generates a logical address (e.g., $346$)
2. The MMU holds a relocation register value (e.g., $14000$)
3. The physical address is calculated as: $Logical\ Address + Relocation\ Register = 346 + 14000 = 14346$
---

### Memory Partitioning & Allocation
#### Memory Partitioning:
Main memory is typically split into two distinct partitions: one for **Kernel processes** and another for **User processes**
- Each partition is contained within a single, continuous section of memory
#### Contiguous Allocation:
- **Sequential Access:** When memory is allocated contiguously, data can be read in sequence, which is significantly faster than non-contiguous access
- **Contiguous Data:** Where data is stored in adjacent, uninterrupted memory locations (one after the other)
- **Management Tracking:** The OS must maintain information regarding which partitions are currently allocated and which are free "holes"
###### Sequential Data Access
- **Definition:** Where the system moves through memory locations in a specific, predetermined order (e.g. like a linked list)
- **Efficiency:** Contiguous memory is read faster because we can fetch the next piece of data without having to calculate complex addresses
###### Sequential vs Random Access Memory:

| **Feature** | **Sequential Access**                       | **Random Access (Direct Access)**                      |
| ----------- | ------------------------------------------- | ------------------------------------------------------ |
| **Order**   | Must follow a strict linear sequence        | Can jump to any address immediately                    |
| **Speed**   | Faster for large, continuous blocks of data | Faster for retrieving small, scattered data points     |
| **Example** | Reading a large file stored contiguously    | Accessing a specific variable in a non-contiguous heap |

---
### Beyond Spec Info (For Fun?)
##### Sequential Access & Predictability:
- **Program Design:** Many common computing tasks are naturally linear, where the code is designed to access the next piece of data immediately after the current one (E.g. iterating through an array or reading a large file)
- **Spatial Locality:** The tendency of a computer program to access memory locations that are geographically close to recently accessed locations within a short time frame
- **Prefetching:** Modern CPUs can exploit this predictability. When the CPU detects a sequential access pattern (e.g. iterating through an array), it "prefetches" the next few blocks of data from main memory into high-speed **cache** before the program even asks for them
##### Link to Contiguous Data:
- This is why **Contiguous Data Allocation** is so fast. Because the data is in one sequence, the hardware can preload it without needing to wait for the OS to find the next data "fragment" scattered elsewhere in the CPU
- Fetching data from main memory is a common bottleneck for execution speed
---
### Memory Hole Allocation (Back to the Spec)
##### Holes in Memory:
- A **hole** is a block of available memory scattered throughout the physical memory space
- **Dynamic Allocation:** When a new process arrives, the OS searches for a hole large enough for it to fit. When a process exits, it frees up the partition, and if this leads to adjacent free spaces, then they are combined into one larger hole
---
### Memory Hole Allocation Strategies:
To satisfy a request of size $n$ from a list of free holes, the OS can use several strategies:
#### 1. First-Fit:
- **Approach:** Allocate the very first hole found in the list that is large enough
- **Benefit:** Generally the fastest strategy as it stops searching as soon as a match is found
#### 2. Best-Fit:
- **Approach:** Allocate the smallest hole that would still fit the process
- **Requirement:** OS must search the entire list unless the list is pre-ordered by size
- **Benefit:** Produces the smallest possible "leftover" hole, which optimises storage utilisation
#### 3. Worst-Fit:
- **Approach:** Allocate the largest available hole
- **Requirement:** Have to search the entire list
- **Benefit:** Leaves behind the largest possible leftover hole, which may be more useful for future large processes than the tiny fragments left by Best-fit
---
### Fragmentation
#### External Fragmentation:
- Where there is enough total free memory space to satisfy a request, but the available space is not **contiguous,** but is instead broken into many small holes scattered across memory
- **Simple Example:** Imagine you have 5 empty seats in a cinema, but they are all single seats scattered around the room. A group of 3 friends cannot sit together, even though there are 5 seats "free" in total
- **The 50-Percent Rule:** Analysis of the "First-fit" strategy reveals that for every $N$ blocks allocated, $0.5N$ blocks are typically lost to fragmentation, meaning up to $1/3$ of memory may be unusable
---
#### Internal Fragmentation:
- Memory that is successfully allocated to a process but remains unused because the allocated block is slightly larger than the requested amount
- **Cause:** Memory is often allocated in fixed-sized blocks (e.g. powers of 2 like 4, 8, or 16 bytes)
- **Simple Example:** If a process requests **27 bytes**, the system might allocate a **32-byte block**. The **5 bytes** of "leftover" space inside that block cannot be used by anyone else and is wasted
---
#### Compaction (Potential solution):
- A technique used to reduce external fragmentation by shuffling memory contents so that all free memory is places together in one large, continuous block
- **Constraint:** This is only possible if relocation is dynamic and done at execution time using relocation registers
---
## Paging
- **Paging** is a memory management scheme that eliminates the need for contiguous allocation of physical memory
- The approach avoids **external fragmentation** and the problems associated with fitting varying-sized memory chunks into holes
---
#### Definitions:
###### Frames:
- Fixed-sized blocks that divide **physical memory**. Sizes are typically a power of 2, ranging between 512 and 8192 bytes
###### Pages:
- Equally sized blocks that divide **logical memory** (Logical meaning that its what the CPU sees when it looks at a program's code and data)
###### Page Table:
- A data structure used to manage the mapping (translation) of logical addresses to physical addresses
